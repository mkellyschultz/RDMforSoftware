Research Data Management for Software/Models

 

Version Control System (or Revision Control System)

·         Software is a special type of data and should be developed with versioning. Version Control systems manage changes to a group of files that usually make up a piece of software (but the concept of saving versions of your files and keeping track of changes is useful for all data). This system tracks what changes are made, by whom, and for what purpose.

·         It is useful, especially when working collaboratively, to make sure that no changes are overwritten or lost or conflicting with each other, and so that versions of files can be tracked, with the potential to compare or revert back to a previous version.

·         This type of system is useful when working on live code/data, but it can also be used to provide a snapshot/release/version to be archived for long-term preservation.

·         “Sometimes there is a good reason to preserve a particular version of some software, in particular, when it has been used to produce other (non-software) data that is published as an academic research output (with or without an accompanying peer reviewed research article). That version of the software and the documentation that would help someone recreate the data are key metadata that can be used in further research and for verification.” - Andy Turner (JISC Research Data Management Discussion List)

·         Some popular open source version control systems:

o   Git: http://git-scm.com/

o   Subversion: http://subversion.apache.org/

o   Mercurial: http://mercurial.selenic.com/

·         Some online sites that provide version control systems (and other services):

o   Github: https://github.com/

o   SourceForge: http://sourceforge.net/

o   Google Code: http://code.google.com/

·         Version control systems records are also very helpful when dealing with issues of ownership and licensing, by keeping track of who wrote what, etc.

·         More information on version control can be found on the OSS Watch Briefing Note on Version Control Systems: http://oss-watch.ac.uk/resources/versioncontrol

·         A helpful book on this topic:  http://ericsink.com/vcbe/

Intent to Share Code (live or archival version) or any data for that matter

·         Why would you want or need to share your live or archival versions of your code?

o   Some funders require you to make any data (including models or code) underpinning your research available/accessible

o   If you want to use some sites, such as Github, you are asked to make your code public (unless you pay for private repositories)

o   You might want to share your code, so that others will be able to reproduce and verify your research

o   You might want to share your code, to allow others to expand and improve it and get input from the community and your peers

o   You might want to share an archival version of your code so that you can then cite it in publications via a DOI or URL

Licensing

·         When planning to share code, it is important to start thinking about licensing (i.e., what you want others to be able to do with your code): http://oss-watch.ac.uk/resources/iprguide

·         If you want to apply a licence and make your code open source there are a number of ready-made options that you can use to apply to your code, handling the legal aspects for you. However, with all software you create as part of your research, the rights are actually owned by the University, so you would have to contact your department head or research services to get permission to license and share you code

·         If licensing is new to you, you might want to start by reading this helpful paper on software licenses by OSS watch: http://oss-watch.ac.uk/resources/licdiff

·         OSS watch also has a helpful tool to compare various open source licenses: http://oss-watch.ac.uk/apps/licdiff/

·         If you want to patent or commercialise your research data, including your software, you might want to talk with Isis Innovation Limited (an Oxford spin-off company that provides this type of help and advice): http://isis-innovation.com/about/ or you might want to consider dual licensing options. Read this article from OSS Watch for more details: http://oss-watch.ac.uk/resources/duallicence2

·         Keep in mind that some licenses may be useful for other data, such as Creative Commons licenses (http://creativecommons.org/licenses/), but they are not intended for software, so another license must be chosen in that case

 

Metadata/Documentation/Commenting Code

·         When considering how much time to spend on creating metadata (which is data about data, so in order words your documentation, README files, comments in code, etc. ), you need to consider the purpose. Will you be sharing your code and want others (or yourself a few years in the future) to run it, modify it, enhance it, understand it? If so, documentation will be essential.

·         There are different things to consider with this: internal documentation and external documentation.

·         Internal documentation might mean commenting your code, using meaningful names for variables/functions/classes, etc. Related to this, you might design your code with extendibility in mind, structuring it so that it can be easily understood and expanded.

·         External documentation might be describing what hardware and software are required to run your code – a description of the environment – to allow others to run and reproduce what you’ve done. If you’re sharing your code and it becomes a living organism, you might have to set up a forum or mailing list and support contact points to manage this code project, including communication of bug fixes, patches, new releases, feature requests, etc. Even if your code is more of a finished product, other researchers might at the very least need to know who to contact if they have any questions about it.

·         There are many coding best practices on the web, including project style guides that might be helpful to adopt for your own coding projects. Some examples include:

o   https://ace.apache.org/dev-doc/coding-standards.html

o   http://journal.frontiersin.org/Journal/10.3389/fgene.2014.00199/full

 

Sharing Live Code/Data

·         Some online sites allow you to share live code, provide version control systems (as mentioned before), and provide forums, issue trackers, etc. to manage a live code project collaboration. Some major examples include:

o   Github: https://github.com/

o   SourceForge: http://sourceforge.net/

o   Google Code: http://code.google.com/

·         If a number of you and your colleagues are using or plan to use Github, it might be useful for you as a group to set up a collection to collect all projects from a research group or department. This is a great way to showcase and communicate current projects.

·         You might want to share whole models, but keep in mind that if you do so, you will need to provide more information, such as what data goes into the model and what data is generated out from the model, the model itself, along with the model framework needed to run the model, along with details about the external environment needed to run this model.

Longer-term preservation, archiving, and sharing Code/Data

·         Github has built-in functionality to deposit a snapshot of your code project into data repositories for longer-term preservation and archiving. Currently, Github supports deposit into figshare and Zenodo. More information from Github can be found here: https://guides.github.com/activities/citable-code/

·         From the Figshare website (http://figshare.com/about): “figshare is a repository where users can make all of their research outputs available in a citable, shareable and discoverable manner. figshare allows users to upload any file format to be made visualisable in the browser so that figures, datasets, media, papers, posters, presentations and filesets can be disseminated in a way that the current scholarly publishing model does not allow.”

·         From the Zenodo website (http://zenodo.org/about): “ZENODO builds and operate a simple and innovative service that enables researchers, scientists, EU projects and institutions to share and showcase multidisciplinary research results (data and publications) that are not part of the existing institutional or subject-based repositories of the research communities.

ZENODO enables researchers, scientists, EU projects and institutions to:

o   easily share the long tail of small research results in a wide variety of formats including text, spreadsheets, audio, video, and images across all fields of science.

o   display their research results and get credited by making the research results citable and integrate them into existing reporting lines to funding agencies like the European Commission.

o   easily access and reuse shared research results.”

·         These are just two examples of repositories to consider for the long-term storage and preservation of your research data, including software. A more comprehensive list of data repositories can be found here: http://www.re3data.org/ Just package up your files/data and submit it with your documentation/metadata.

·         If you don’t feel you have an appropriate subject-specific repository for your software/data, a team of staff from the Bodleian Libraries at the University of Oxford are currently setting up an institutional data repository called ORA-Data that you might want to consider. The pilot launches December 1. For more information: http://researchdata.ox.ac.uk/preserving-your-data/ora-data/

·         In addition, many data repositories (including figshare, Zenodo, and ORA-Data) provide a persistent identifier for your data (i.e., a DOI) so that you can cite your data in publications or share a link to your data with colleagues.

·         Data or software journals might be a complementary way to share your archived data. A helpful list of suggestions can be found here on the SSI website: http://www.software.ac.uk/resources/guides/which-journals-should-i-publish-my-software

 

Software Preservation Approaches

·         Based on the preserving and curating software document from OSS Watch: <URL>

o   “Technical preservation (techno-centric) - Preserve original hardware and software in same state

o   Emulation (data-centric) - Emulate original hardware and operating environment, keeping software in same state

o   Migration (functionality-centric) - Update software as required to maintain same functionality, porting/transferring before platform obsolescence

o   Cultivation (process-centric) - Keep software ‘alive’ by moving to more open development model bringing on board additional contributors and spreading knowledge of process

o   Hibernation (knowledge-centric) - Preserve the knowledge of how to resuscitate/recreate the exact functionality of the software at a later date

o   Deprecation - Formally retire the software without leaving the option of resuscitation/recreation

o   Procrastination - Do nothing”

·         Guidelines can be found from an STFC project on identifying significant properties of software: http://bit.ly/eF7yNv
